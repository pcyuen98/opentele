SWAGGER UI FEATURES
================================================================================

1. API Overview
   - Title: OpenTelemetry Stacktrace Tracker API
   - Version: 1.0.0
   - Description: API for tracking and persisting stack traces with 
     OpenTelemetry using Redis

2. Available Endpoints (visible in Swagger UI):
   
   a) GET /api/health
      - Health Check
      - Tests if service is running
      - No parameters required
      - Response: {"status":"UP", "message":"OpenTelemetry Stacktrace 
        Tracker is running"}

   b) POST /api/track-error
      - Track Error with OpenTelemetry
      - Parameter: message (required, string)
      - Example: POST /api/track-error?message=TestError
      - Response: Tracked error with timestamp, IP, stacktrace details
      - Data is persisted to Redis

   c) GET /api/tracked-stacktraces
      - Get All Tracked Stack Traces
      - Retrieves all stack traces from Redis
      - No parameters required
      - Response: {"count": N, "stacktraces": [...]}

   d) GET /api/tracked-stacktraces/ip/{ip}
      - Get Stack Traces by IP
      - Parameter: ip (path parameter, e.g., 192.168.1.1)
      - Response: Stack traces only from specified IP

   e) DELETE /api/clear-stacktraces
      - Clear All Stack Traces
      - Removes all data from Redis
      - No parameters required
      - Response: {"status":"cleared", "message":"All tracked stacktraces 
        have been cleared from Redis"}

   f) GET /api/documentation
      - Get Documentation Page
      - Returns HTML documentation
      - Viewable in browser

REDIS DATA PERSISTENCE
================================================================================

Data is automatically persisted to Redis with the following structure:

Keys:
   - stacktrace:{trace_uuid}        : Individual trace data
   - stacktraces:index              : Set of all trace IDs
   - stacktraces:ip:{ip_address}    : Set of trace IDs by IP

Verify data in Redis CLI:
   redis-cli
   > KEYS stacktrace:*               # View all stack trace keys
   > SMEMBERS stacktraces:index      # View all trace IDs
   > GET stacktrace:{id}             # View specific trace data

Data persists across:
   - Application restarts (stored in Redis)
   - Server restarts (depends on Redis persistence settings)

Clear all data:
   redis-cli FLUSHALL

OPENTELEMETRY INTEGRATION
================================================================================

OpenTelemetry Tracing:
   - Each error tracking creates a span: "track-stacktrace"
   - Span attributes:
     * ip: Client IP address
     * error.type: Exception class name
     * error.message: Exception message
   - Spans exported to logging

View spans in logs:
   - Application logs show all OpenTelemetry spans
   - Set logging level: logging.level.io.opentelemetry=DEBUG

TROUBLESHOOTING
================================================================================

1. Build Failures:
   - Ensure Java 21: java -version
   - Clear Maven cache: mvn clean
   - Check JAVA_HOME is set correctly

2. Tests Failing:
   - Redis may not be running (tests handle this gracefully)
   - Clear Maven cache: mvn clean

3. Application Won't Start:
   - Port 5080 may be in use: change with --server.port=5081
   - Check logs for errors
   - Verify Java 21 is being used

4. Swagger UI Not Loading:
   - Verify application is running: curl http://localhost:5080/api/health
   - Check browser console for errors
   - Verify port 5080 is accessible

5. Redis Connection Issues:
   - Verify Redis is running: redis-cli ping
   - Check Redis host/port in application.properties
   - Application works without Redis (falls back to in-memory)

6. No Data in Swagger Response:
   - Redis may not be running
   - Check application logs for connection errors
   - Data is still tracked even if Redis is unavailable
